## 🚨 Launch blockers

- [x] **`admin_insert_shipment` RPC fails on every label call — shipments have not been persisted to the DB since March 18** — 🚨 **LAUNCH BLOCKER. CRITICAL.** Discovered 2026-05-12 during Phase A smoke test (UPS Ground `1Z13J52C0333598579`). Event log evidence: only ONE `label.db_persisted` row exists in `event_logs` across the entire history (2026-03-18). Every label since then has been bought successfully from EasyPost but **never written to the `shipments` table.** Two compounding root causes: (1) **Function overload collision** — production has TWO `admin_insert_shipment` overloads sitting alongside each other. Migration 014's `DROP FUNCTION IF EXISTS` targeted the 29-param signature it expected from migration 012, but migration 012 may not have applied cleanly to prod, so the DROP was a no-op and the new TABLE-returning version landed as a sibling to the surviving UUID-returning version from migration 008. PostgREST function lookup can't resolve which overload to call. (2) **Frontend address shape bug** — `addressToApi` in `src/lib/api.ts` maps `addr.street → street1`, but at runtime `addr.street` is `undefined` despite the `AddressInput.street: string` type claim. JSON.stringify drops undefined values, so `from_address.street1` and `to_address.street1` are missing from the labels function call, which causes the RPC param lookup to fail against the version that has those as required-no-default params. Fix has three parts: (a) Write migration 018 that explicitly drops both overloads and recreates the canonical TABLE-returning version, (b) trace where `addr.street` becomes undefined in the flow (likely `RecipientStepFullShipping` or the smart-address-input wiring) and make the field actually populated, (c) consider tightening `addresses.street1` constraint enforcement so future failures are visible at the function call site instead of bubbling up as PostgREST 404s. **Until fixed, every Full Label flow generates an EasyPost label without a corresponding shipments row** — meaning the admin report, dashboard, /admin reconciliation, webhook tracker.updated lookups, all of it has no shipment to join against. Phase A's `comp_grant` smoke test exposed this; Phase A is not the cause.

- [x] **EasyPost webhook HMAC verification rejecting all events** — 🚨 **LAUNCH BLOCKER.** Nine consecutive `webhook.hmac_invalid` events in `event_logs` with `reason: signature_mismatch` between 22:49 and 22:54 on 2026-05-12. EasyPost is POSTing `tracker.updated` events (and likely others) to `webhooks/index.ts`, but our HMAC validation is failing every signature check. Result: no tracking-update DB writes, no notification dispatches on delivery, every shipment's status stays at `label_created` forever. Likely root cause: `EASYPOST_WEBHOOK_HMAC_SECRET` configured in Supabase doesn't match the secret EasyPost is signing with (rotation drift, wrong project's secret, or initial setup mismatch). Fix: (a) confirm the secret in Supabase function env via `npx supabase secrets list --project-ref fkxykvzsqdjzhurntgah`, (b) reconcile against the value in the EasyPost dashboard webhook settings, (c) rotate both sides if they've drifted. Until fixed, no shipment transitions to `in_transit` / `delivered`, and the WISHLIST "Tracking status stuck" bug regresses despite the prior `tracker.updated` URL registration.

- [x] **Share Link on Label & Link step is hardcoded `sendmo.co/s/test`** — 🚨 **LAUNCH BLOCKER.** After a Full Prepaid Label flow completes (Label & Link step), the Share Link card surfaces a placeholder URL (`sendmo.co/s/test`) instead of a real shareable link tied to this shipment. The Full Label path doesn't write a `sendmo_links` row, so there's no real `short_code` to surface. Recipients can't share the label with anyone today. Confirmed live 2026-05-12 — see UPS Ground `1Z13J52C0333598579` Phase A smoke-test screenshot. Two-part fix: (1) labels function (or a new step) writes a `sendmo_links` row with a generated `short_code` after the shipment lands, owned by the verified recipient; (2) RecipientStepLinkReady reads the real `short_code` and displays `sendmo.co/s/<code>`. Until this lands, the Full Prepaid Label flow is end-to-end demo-able but not actually launchable. Related: the existing "Full Label flow doesn't create account or link" item below covers the account-creation half (which shipped 2026-05-11 via the OTP rework); the link-creation half is what remains. **Cross-link:** the persistence bug above means there's no shipment row to attach a `sendmo_links` row to anyway, so the persistence fix is a prerequisite to this one.

### Existing Features (In Progress)
- [ ] **Failure-mode tracking emails — void/refund, return_to_sender, exceptions** — Today's notification pipeline only fires emails for the happy-path statuses (`in_transit`, `out_for_delivery`, `delivered`). Silent failure modes that the user never hears about: (1) **Label voided / refund processed** — recipient was told the package is on its way (label-confirmation email already sent), then the label is voided; they never learn it's not coming. Send a void email to both sender + recipient when `refund_status` transitions to `submitted` or `refunded`. (2) **Return to sender** — `shipments.status = 'return_to_sender'`, currently a silent DB state. Send an "Your package is being returned" email with the reason from the latest tracking event if available. (3) **Exceptions** — EasyPost surfaces non-terminal exception events (address invalid, delivery attempted, held at facility). Decide whether to forward as a tracking-update email or batch into a single "Delivery exception" email. Implementation: extend `NOTIFY_STATUSES` in [tracking/index.ts](supabase/functions/tracking/index.ts) + [webhooks/index.ts](supabase/functions/webhooks/index.ts); add new template functions in [_shared/email-templates.ts](supabase/functions/_shared/email-templates.ts); wire void emails to the existing refund path in [refundService.ts](src/lib/refundService.ts). Surfaced during the 2026-05-11 email audit.
- [ ] **Retry transient Resend failures** — `notifications_log` rows can land at `status='failed'` if Resend returns 5xx or times out, and the next webhook on the same shipment for a *later* status won't backfill the missed earlier one (idempotency check is per-event-type). Add a retry worker — either an Edge Function on a cron, or a Supabase scheduled trigger that re-attempts `status='failed'` rows < 24h old with exponential backoff. Cap retries at 3. Surfaced during the 2026-05-11 email audit.
- [ ] **Cache `tracking_details` events in DB** — Tracking page now always fetches live from EasyPost on every load (no TTL cache; reads are free), so this is lower priority. Worth doing eventually so the events list still renders if EasyPost ever goes down.
- [ ] **Register EasyPost webhook URL** — Configure EasyPost dashboard to send `tracker.updated` events to `https://fkxykvzsqdjzhurntgah.supabase.co/functions/v1/webhooks`. The webhook handler is already built and deployed — just needs the URL registered. This will give us real-time status updates pushed by EasyPost instead of polling on page load. Currently we use a 30-min TTL cache on the tracking function; webhooks would make this near-instant and eliminate API polling entirely.
- [ ] **Cache tracking events in DB** — Store EasyPost tracking events (message, status, datetime, location) in a `tracking_events` table so the tracking page can show event history even when serving from cache (within the 30-min TTL). Currently events only show when a fresh EasyPost fetch happens.
- [ ] **Payment transaction ledger** — Per CLAUDE.md rule 16, financial balance changes must use an append-only ledger table. Create a `transactions` table: `id, user_id, shipment_id, type ('charge'|'refund'|'credit'|'fee'), amount_cents, stripe_ref, description, created_at`. All money movement flows through this table. Required before Stripe goes live.
- [ ] **AI-powered FAQ / Help center** — Replace the static FAQ page with a smart help experience. Include: (1) AI-based search that answers questions from a knowledge base (shipping policies, pricing, how-it-works, troubleshooting), (2) a curated list of real questions people actually ask (sourced from support emails, user testing, etc.), and (3) unified FAQ + Help in one place (accessible from header for both signed-in and signed-out users). Consider embedding an AI chat widget or search-first UX.
- [ ] **Flexible label generation** — After the sender selects a shipping rate on the flex link flow, generate and display the actual shipping label (PDF). Requires: Stripe payment capture on the flex link's authorized hold, EasyPost label buy call with the selected rate, label PDF storage in Supabase Storage, and a download/print UI on the done step. Currently the done step shows a placeholder ("Label generation coming soon").
- [~] **Stripe refund on label void** — *Partially shipped 2026-05-13.* Two-step refund architecture landed: `cancel-label` no longer fires Stripe `createRefund` immediately; `tracking/index.ts` lazy-polls EasyPost on `/t/<code>` page views for shipments in `refund_status='submitted'`, and when EP confirms `refunded`, fires the Stripe refund (or marks comp as `not_applicable`). `stripe-webhook` then advances `refund_status` to `refunded` on `charge.refunded`. Remaining work: (a) verify EasyPost's refund webhook events (likely `refund.successful` or similar) and wire to `webhooks/index.ts` for push-based confirmation — eliminates the page-view dependency; (b) cron-based active polling for stale `submitted` shipments that nobody visits. Today neither is critical because zero Stripe-paid shipments exist (Phase E hasn't shipped); polling-on-page-view covers all real cancels today. Admin-initiated counterpart in [refundService.ts](src/lib/refundService.ts) (`processRefund()`) is still stubbed. See LOG entry 2026-05-13.
- [ ] **EasyPost refund webhook wiring** — Verify exact event names against EasyPost's webhook docs (probably `refund.successful` / `refund.rejected` or bundled into `tracker.updated` with a refund-status field), then add a case to `webhooks/index.ts` to handle them. When the event fires for a shipment in `refund_status='submitted'`: same two-step logic as the tracking-poll path — call Stripe `createRefund` if PI present, mark `not_applicable` if comp, mark `rejected` if EP rejected. Cuts latency from "next page view after carrier confirms" to "seconds after carrier confirms." Won't matter until Phase E produces Stripe-paid cancels; lazy poll covers today.
- [ ] **Cron-poll for stale `refund_status='submitted'` shipments** — pg_cron or a scheduled Edge Function (every 6h) that finds shipments where `refund_status='submitted' AND updated_at < now() - interval '24 hours'` and polls EasyPost for each. Catches the case where a cancellation lands but nobody visits `/t/<code>` and the EP webhook doesn't fire (or is missed). Set up when volume justifies (~100+ cancellations per month); irrelevant at MVP.
- [ ] **User payment history page** — Show all transactions (charges, refunds, credits) for the authenticated user. Read from the `transactions` ledger table. Accessible from Dashboard.

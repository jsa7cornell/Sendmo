Mental model

One page. Three lifecycle states. Layered on top: viewer-identity and referral-source variants that augment the state.

  • Primary axis — lifecycle: pre-drop-off → post-drop-off → post-delivery. Determines the hero, headline, what actions are available.
  • Viewer-identity axis: payer (logged in, owned-link cookie, or signed email link) vs anonymous. Determines whether receipt + cancel surface at all.
  • Referral axis: fresh from checkout (?just=bought) vs returning. Currently used only for an optional confetti flourish on the pre-drop-off hero; the underlying content is the same.
What this replaces: the inline LabelReady in RecipientStepPayment, the parallel sender-flow success view, and the "is this confirmation or tracking?" question. One page. State-driven. Viewer-aware.

v4 design directives — John, post-review

Five new directives from John applied after the fresh-eyes review came back. Each touches a visible element in the mockups above.

A

"ETA banner gets a matching SVG hero in the same visual language as the truck — a person walking with a package."

Done. Replaced the small calendar icon with an illustrated SVG showing a person mid-stride carrying a package, plus motion lines and a dashed ground line — matching the truck hero's compositional grammar (ground line + motion lines + character + box detail). Visible on all three pre-drop-off variants (mockup ①, payer-already-printed, anonymous).

B

"State hero graphic and text 25% smaller."

Done. Scene height 86px → 64px (≈26% reduction). H1 font 17px → 14px (≈18%). The heroes now sit more quietly above the actionable content; the ETA banner mini-hero is the new visual anchor on pre-drop-off.

C

"'Everything OK?' card on delivered is overstated — remove. Make it a 'Need help' link next to cancel (or in the cancel slot if cancel is gone) inside DetailsCard."

Done. The standalone "Everything OK?" card is removed from the delivered state. A "Need help" link now lives in the DetailsCard footer on every viewer + state combination — alongside Cancel when present, alone in the cancel slot when not. Mailto target stays support@sendmo.co; subject/body pre-fill with tracking + from/to + current lifecycle state context so John can triage inbound.

D

"When cancel isn't available, don't say 'cancel unavailable' — just hide it."

Done. The inert grey "Cancel unavailable — package has been scanned" note is removed from mockup ② (in-transit). When cancel is ineligible, the cancel slot is empty; the "Need help" link occupies the row alone (right-aligned). No copy required to explain absence.

E

"How do we handle scheduled pickup vs drop-off? Does the hero update?"

v1: drop-off only — matches today's reality. SendMo has no pickup wiring in the codebase (grep'd; no Pickup calls into EasyPost). All three pre-drop-off mockups assume the shipper physically drops the package off. v1.x or later: pickup adaptation. When EasyPost Pickup API integration ships (wishlist entry added 2026-05-19), the pre-drop-off hero swaps the "person walking with package" illustration for a "carrier arriving at the door" illustration, the ETA banner copy changes from "Drop off today → arrives X" to "Pickup scheduled Wed → arrives X", and HowToShipStrip step 3 ("Drop off") becomes "Leave for pickup." Detection signal: presence of a non-null shipments.pickup_easypost_id (column doesn't exist today; would land with the pickup-API proposal).

F

"Non-paying sender (filled a prepaid link) should land on pre-drop-off but with no receipt. Instead of receipt, say 'X (the recipient) has paid.'"

Done. New sender_flex viewer role — distinct from payer and anonymous. Renders pre-drop-off identically (state hero + ETA banner + print/download + how-to-ship + details with cancel + Need help), but the receipt block is replaced by a soft-green confirmation block: a green check + "Jane has paid for shipping · No charge to you — the prepaid label is on the recipient." See the middle mockup in the variants row above.

Server derivation: in tracking/index.ts, add a viewerRole field computed from (viewerIsRecipient ? "payer" : viewerHoldsCancelToken ? "sender_flex" : "anonymous") where viewerHoldsCancelToken means the caller passed a valid cancel_token matching shipments.cancel_token. That's the same auth proof we already use for the sender's cancel surface (decided in 2026-05-11_label-cancel-and-change) — so the sender's "I just shipped this" identity is provable without new infrastructure. Recipient first-name surfaces via the existing sendmo_links.user_id → profiles join (already loaded server-side; need to add first_name to the SELECT). Privacy: sender already knows the recipient — they just shipped to them — so revealing first name is not a leak.

v3 nit responses (preserved for institutional memory)

1

"Drop the PREVIEW tearsheet — keep the SVG state visual instead. Stick with the inline SVGs for v1."

Done. Tearsheet block + watermark + caption removed from every pre-drop-off variant. The "labelness" of the moment now lives entirely in the SVG illustration at the top — a package with a label affixed and sparkles. No risk of mistaking a thumbnail for a printable artifact, and one less component to maintain. (We can revisit polishing the SVG illustrations as a family if/when the rest of the product gets an illustration pass.)

2

"Add a prominent ETA banner — 'if you get it to {carrier} today, expected delivery around {date}'. Smart about what day it is."

Done. A blue-tinted banner sits right under the hero, on every pre-drop-off variant: calendar icon + carrier-aware copy. Today is Tue May 19; USPS Ground Advantage estimated 3 days; Mon–Sat delivery → "Drop off at USPS today → arrives Sat, May 23." Smaller subline gives the cutoff context ("drop by 5pm today for next-day pickup").

Logic lives in a new helper computeDeliveryEta(carrier, estimatedDays, now) returning { deliveryDateLabel, cutoffNote }. Inputs: carrier name (already in labelResult.carrier), labelResult.selectedRate.estimated_days, and new Date(). The helper:

  • Knows each carrier's delivery schedule: USPS Mon–Sat, UPS/FedEx Mon–Fri, DHL Mon–Fri.
  • Knows pickup cutoffs (default 5pm local; configurable per-carrier in a constants table).
  • If "now" is past the cutoff, copy shifts to "Drop off tomorrow (Wed) → arrives …".
  • If "now" is on a non-pickup day (Sun for USPS retail, weekend for UPS/FedEx), copy shifts to "Drop off Mon → arrives …".
3

"Use the existing print counter; just keep the button thinner. Don't reinvent."

Done. No new schema, no new state. Existing server-side shipments.print_count + /label-print endpoint are reused. The visual treatment changes:

  • Buttons stay equal-width and thinner (padding reduced from 10px to 8px vertical). No inline chip making one wider than the other.
  • Below the button row, one small centered line: "Not printed yet" when count is 0, "✓ Printed 1× · tap again to reprint" when count ≥ 1 (success-green).
  • The Print button itself gains a subtle "done" styling once print_count > 0 — soft green tint, no shape change.
  • How-to-ship step 1's numbered circle becomes a green checkmark on first print, so "I did that" reads from two places without doubling the noise.

Download isn't tracked (no download_count column). Punt: localStorage hint, or just leave Download visually static. Author leans static — Print is the meaningful business event.

4

"Receipt at bottom, below delivery information. Show on payer just-bought OR payer returning. Anonymous shows nothing — no link, no modal. (Email-receipt idea → wishlist.) Payers can always reach the receipt from their dashboard."

Done. Receipt block now lives at the bottom of the scroll, beneath the Shipment Details card, on every payer view. Density varies:

  • Payer + just-bought — full receipt (Shipping line, Insurance line, Charged total, payment method, timestamp + PDF link).
  • Payer + returning — condensed single-line receipt ("$7.59 · charged to •••• 4242 · May 19") + PDF link.
  • Anonymous — no receipt block, no "View receipt" link. Payment information is entirely absent from anonymous views. Payers who want the receipt sign in via Dashboard → tracking page renders the full version.

The "email the receipt to me" modal is wishlisted (good idea, not v1).

5

"Contact support = mailto: for now. Wishlist a real support system."

Done. Delivered state renders an "Everything OK?" card with a single "Email support" link that opens mailto:support@sendmo.co with a pre-filled subject (Issue with shipment <tracking>) and a body that includes the tracking number, from/to cities, delivered timestamp, and an empty "What happened:" field. Address confirmed at src/pages/Terms.tsx. Wishlist item added for a proper support intake (ticket system, in-app form, or Zendesk integration).

Open questions for reviewer

"Payer" gate signals
Proposal counts the viewer as payer if any of: (a) auth session userId matches link owner, (b) recent post-payment session cookie set for this shipment, (c) signed query param from the receipt/label email (?as=payer&t=…). Default falls back to anonymous. Are these the right three? Anything missing? Author stance: a + b + c. Conservative — anonymous unless we have proof.
ETA helper edge cases
Three edge cases worth deciding now: (1) Late-night usage — at 11pm Tue, do we say "today" or "tomorrow"? Proposal: past 5pm = "tomorrow". (2) Federal holidays — currently ignored; conservative path is to add USPS observed holidays as a constants table. (3) estimated_days missing — fallback to a carrier-default range or hide the banner? Author stance: cutoff = 5pm; holidays added in v1 as a static list; hide banner if estimated_days is null.
Sender-flex parity
Senders who finish flex checkout also land on a fresh label. Today SenderFlow pushes a cancel token via the same query-param transport and routes to /t/<code>. Adding ?just=bought to that redirect gives senders the same celebration moment. They show as payers (they paid), so receipt-at-bottom + cancel-in-details all apply. Author stance: include senders. Same surface, same rules.
"Cancel" placement when cancel is unavailable
Today's mockup shows an inert grey note ("Cancel unavailable — package has been scanned") in the Details card for payer+in-transit. Alternative: hide the slot entirely. Author leans showing the note — it answers "why isn't the cancel button there anymore?" without making the user hunt. Author stance: show the inert note.
e2e suite impact
The full-prepaid Playwright spec currently asserts on the inline LabelReady view. Those asserts need to follow the redirect to /t/<code>. Estimate ~3 specs to update. Pre-merge audit task: grep tests/e2e/ for "Your shipping label" and "Print Label" inside the recipient-onboarding suite. Action: pre-merge audit of e2e specs.

Trade-offs & risks

Variant complexity. 3 lifecycle states × 3 viewer identities × 1 referral flag = up to 9 distinct rendered combinations. Mitigation: the viewer axis only swaps three blocks (cancel surface, receipt block, "Everything OK?" link). Lifecycle is the heavy axis with three branches. Manageable.
SVG illustrations are crude. The three inline SVGs in this mockup are functional but not portfolio-ready. They'll feel "design-system-y in the bad way" until a polish pass. Author recommends shipping v1 with them as-is and revisiting once the rest of the product gets an illustration pass.
ETA helper accuracy. "Drop off today → arrives X" is a promise. If our cutoff/holiday logic is wrong, we'll under- or over-promise. v1 mitigation: use phrasing like "expected delivery around" (already proposed) and add a small "* estimated" tooltip on the date. v2 could surface EasyPost's delivery_date_guaranteed when available.

Implementation sketch

  1. RecipientStepPayment.tsx & SenderFlow success branch — replace inline success view with navigate(\`/t/\${publicCode}?just=bought\`, { replace: true }). Cancel token continues via existing query-param transport.
  2. TrackingPage.tsx — derive lifecycleState from shipment status (label_created → pre-drop-off; in_transit / out_for_delivery → post-drop-off; delivered → post-delivery). Derive viewerIdentity from existing auth signals + cookie + email-link param. Render <StateHero kind={lifecycleState} /> + state-specific blocks.
  3. New components in src/components/tracking/:
    • StateHero — SVG scene + state-specific headline.
    • EtaBanner — pre-drop-off only. Consumes computeDeliveryEta() helper.
    • ReceiptBlock — payer-only, two density modes (full vs condensed). Rendered at the bottom of the scroll on every payer view.
    • ContactSupportCard — delivered state. mailto:support@sendmo.co with pre-filled subject + body from shipment context.
  4. New helper src/lib/deliveryEta.ts — computeDeliveryEta(carrier, estimatedDays, now). Constants table for carrier delivery schedules + cutoffs + USPS holidays. Unit-tested.
  5. ShipmentLabelSection — collapse into ActionButtons (the thin Print/Download row + print-count line) + reuse HowToShipStrip. Existing cancel surface moves into DetailsCard as the destructive link. Print-count chip is removed from inside the button.
  6. HowToShipStrip — accept an printDone prop; step 1 numbered circle becomes green check when true. Step 3 number → map pin icon (already drafted in v1's nit response).
  7. e2e: update existing inline-confirmation specs to follow redirect. Add new specs for state-hero rendering, ETA banner copy across days/cutoffs, payer/anonymous variant branches.
  8. Wishlist: "Email me the receipt" modal (deferred — not v1). Real support system (deferred — mailto for v1).

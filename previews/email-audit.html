TL;DR

SendMo has 11 cataloged transactional emails across auth, label lifecycle, tracking, refunds, risk, and admin. The two types confirmed firing in the last 30 days are label confirmation and tracking updates (in_transit / out_for_delivery / delivered). The H5 refund lifecycle emails (A/B/C), risk-intel emails, and carrier adjustment email are all wired but have not yet fired against live shipments.

  • The OTP/magic-link email is handled by Supabase Auth via Resend SMTP — not the custom template. The template exists but has no wired send-site.

2 Why H5 Emails Did Not Fire for YPPY9AK

Root Cause: UPS Rejected Both Void Attempts (Carrier-Side, Not a Code Bug)

The event_logs tell the full story. H5 emails A/B/C are all gated on the cancel path succeeding at the carrier level. For YPPY9AK, the carrier rejected both void attempts — so Email A's prerequisite (refund_status='submitted') was never written, and the entire H5 email chain never started.

YPPY9AK ($9.18 UPS DAP — 2026-05-24)
18:18 UTCOK — Label purchased. E2 confirmation fired (this run pre-2026-06-27 logged email.label_confirmation_sent; since 2026-06-27 E2 routes through the dispatcher and logs notification.email_sent with properties.event = "label_created"). notification_contacts row inserted (count=1).
18:32 UTCLabel printed by admin from print queue.
18:46 UTCCARRIER REJECT — First cancel attempt. cancel.carrier_rejected logged: "UPS returned the error message: Void not available at this time. Please try again later." No refund_status update. No Email A.
18:54 UTCCARRIER REJECT — Second cancel attempt. Same UPS rejection. Still no refund_status update.
19:28 UTCWEBHOOK MISS — EasyPost refund.successful arrives with shape (b) payload (result.id='rfnd_dcda5a...'). Pre-076bf75 bug: the webhooks fn reads result.id as the EasyPost shipment ID (it's actually a refund ID), fails lookup, logs webhook.easypost_refund_shipment_not_found, 200s. No Stripe refund initiated, no Email A/B.
19:59 UTCUNSTUCK — Admin curl against tracking endpoint for YPPY9AK triggers lazy-poll path. EP returns refund_status='refunded'. Stripe refund initiated (pi_2TagbWxS6gsndgF32zGP8SHp). EasyPost refund ledger row written.
19:59 UTCEMAIL B MISS — stripe.charge_refunded fires in stripe-webhook with shipment_id=NULL (pre-44da65e bug: resolvePiContextWithFallback not yet deployed). Email B gated on shipment resolution → skipped.
20:05 UTCSubsequent tracking polls: Stripe refund idempotency key reused → cancel.stripe_refund_failed (expected — refund already issued). No new email triggers.
20:06 UTCOK — Delivery webhook fires for a different shipment. YPPY9AK refund_status now manually set to 'refunded' per LOG guidance.

Three bugs compounded:

  1. c01334e (TDZ crash) — This was fixed before the first live cancel; the event_logs show no TDZ error for YPPY9AK. This bug was NOT a contributing factor for this specific shipment.
  2. 076bf75 (webhook payload-shape miss) — EasyPost's shape-(b) refund.successful was misidentified as a shipment lookup. This killed the push-path unstick. Root cause for the 35-min delay.
  3. 44da65e (PI↔shipment Path B miss) — When the lazy-poll triggered the Stripe refund, charge.refunded couldn't resolve shipment_id from stripe_intents (null by design) and skipped Email B. Root cause for Email B never firing.

Email A specifically: Never triggered because both UPS voids were rejected — refund_status was never written as 'submitted'. This is correct behavior: Email A is intentionally gated on a successful void. The underlying issue is that YPPY9AK was a carrier-rejection, not a code bug in Email A's path.

Both 076bf75 and 44da65e are now deployed. The next live cancel-and-refund cycle should fire Email A (if void succeeds), B, and C correctly.

3 Gap Analysis — Emails SendMo Should Arguably Send
ℹ
Gaps below are rated on two axes: urgency (security / customer-experience / nice-to-have) and timing (pre-launch vs. post-launch). John makes the final calls.
G1 — Welcome / First-Label Email
High — CX

No welcome email exists. The first time someone creates a SendMo link or buys their first label, they get no "here's what you just set up" message. This is a standard onboarding touchpoint that sets expectations, explains the dashboard, and gives support contact info.

Trigger candidates: (a) first link created, (b) first label purchased, (c) account created (see G2). Option (b) is the highest-intent moment.

Pre-launch recommendation: At minimum, a post-first-label email: "Your first SendMo shipment is on its way — here's how to use your dashboard, track it, and what happens next." 1-2 hours of work. Dedup: check shipment_count = 1 at send time.

G2 — Branded OTP / Account-Created Email
High — Brand

Auth OTPs are delivered by Supabase's built-in mailer via Resend SMTP — but using Supabase's default template design, not the custom otpEmail template in email-templates.ts. Users see generic Supabase branding during signup. The custom template exists but has no current send-site (template is orphaned).

Fix: Configure a Supabase Auth [auth.hook.send_email] hook (a Deno edge function) that receives the OTP and sends it via the custom otpEmail template through Resend. This is a ~1hr task. Without it, every new user sees "Supabase" during signup, not "SendMo."

Pre-launch recommendation: Yes. Brand trust during signup is foundational. The template is already written — it just needs the hook wire-up.

G3 — Password Reset / Sign-In Magic Link
Medium — Auth UX

SendMo uses passwordless auth (OTP + Google OAuth). There is no separate "password reset" because there are no passwords. Magic links are sent by Supabase's built-in mailer when signInWithOtp is called with shouldCreateUser: false.

Status: Handled by Supabase. Same branding gap as G2 — Supabase default template. The fix is the same auth hook approach.

Pre-launch recommendation: Subsumed by G2. One auth hook handles both OTP and magic-link scenarios.

G4 — Email Change / Sensitive Action Confirmation
Medium — Security

When a user changes their email address, Supabase Auth handles the confirmation email automatically (sends to the new address). No custom handling exists or is needed at the app layer.

Status: Handled by Supabase Auth's built-in flow. Same Supabase branding gap as G2/G3. No gap in security logic — the confirm-before-switch behavior is correct. Branding fix is the same hook approach.

Pre-launch recommendation: Subsumed by G2 auth hook. Not a separate pre-launch item.

G5 — Payment Method Added Notification
High — Security

When a customer saves a card (SetupIntent via /payment-methods), no email is sent. Industry standard: "A new card ending in XXXX was added to your SendMo account. If this wasn't you, contact us immediately."

Why it matters: SendMo's flex link model means a saved card is charged off-session whenever someone uses the link. If an attacker gains account access and saves a new card, the account holder has no way to know until they see the charge. This email is a standard account-security notification.

Pre-launch recommendation: Yes. Security-tier email. Low effort — fire from payment-methods/index.ts after successful SetupIntent confirm. No custom template exists; needs to be written.

G6 — Payment Method Removed Notification
Medium — Security

No email when a saved card is removed (DELETE /payment-methods/:pm_id). Same security rationale as G5 — if someone removes a card without the owner's knowledge, they'd want to know.

Pre-launch recommendation: Nice-to-have. Lower priority than G5 (removal is less dangerous than addition). Post-launch acceptable.

G7 — Stripe-Branded Receipt at Charge Time
Medium — CX

Stripe can send an automatic receipt email when a PaymentIntent is captured if receipt_email is set on the PI. SendMo passes receipt_email to the payments function for full-label flows. It's unclear whether Stripe receipts are actively enabled on the account.

The current situation: SendMo's label confirmation email (E2) fills this role — it fires after label purchase and includes the amount paid. It's a branded alternative to a bare Stripe receipt. Whether Stripe's unbranded receipt is also sending is not auditable from this codebase (it's a Stripe dashboard setting).

Pre-launch recommendation: Verify in Stripe Dashboard whether automatic receipts are enabled. If yes, decide whether to disable them (to avoid double-email) or customize the Stripe receipt template. If no, E2 is the de-facto receipt — confirm it always includes the price (currently gated on displayPriceCents != null).

G8 — Sender (Link Owner) Tracking Notifications
Medium — CX

The dispatchNotifications system (E3/E4/E5) supports a role: "sender" in notification_contacts. The labels/index.ts stores a sender contact if sender_email is provided in the request. The email template has sender-aware copy ("The package you sent is In Transit").

Gap: In practice, the flex link flow doesn't currently collect or pass a sender email at label-buy time — the "sender" is the anonymous person using the link, who may or may not have an email. Only one or zero contacts are typically stored. This is not a bug per se, but the link owner (payer) has no systematic way to track all shipments made against their link unless they check the dashboard.

Pre-launch recommendation: Post-launch. The label confirmation email (E2) already notifies the link owner on label creation. A full shipped/delivered loop for the link owner is a product-polish feature.

G9 — Carrier Void Rejection Customer Notification
High — CX

When the carrier rejects a void request (as happened with YPPY9AK — "UPS Void not available at this time"), the customer is not notified. Email A is gated on refund_status='submitted' — so a failed void gets no email at all. The UX from the customer's perspective: they requested a cancel, nothing happened visually, and they have to check the dashboard or wait indefinitely.

Contrast: E12 (Refund Unsuccessful) covers the case where the void is submitted but then rejected by the carrier post-submission. But there's no email for the case where the void is rejected immediately at the API call level.

Pre-launch recommendation: Worth considering. A short "we tried to void your label but [carrier] rejected it — we'll keep trying or you can contact us" email would prevent confusion in the YPPY9AK scenario. Low-effort template + single send-site in cancel-label/index.ts.

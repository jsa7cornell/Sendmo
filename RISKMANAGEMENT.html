SendMo — Risk Management

Last updated 2026-05-23 · Scope: payments fraud + abuse · Status: v1 shipped (commits 397530c, dbf2254, 60e09b3)

TL;DR. SendMo's flex-link product lets an anonymous sender charge a recipient's saved card off-session. That's the novel risk surface, and the one we defend ourselves — Stripe Radar can't see the sender on an off-session merchant-initiated charge. Three controls handle it: an Account Budget ($200/day + $500/week per account, admin-raised), a per-account PM-add breaker (5 card-adds/day), and a distinct Radar-block branch in the webhook so a fraud block isn't treated as a card decline. Per-shipment cap ($50 default) and Stripe Radar handle the easier on-session surfaces. The Account Budget is the only customer-visible control — everything else is transparent to good users.

1. Architecture

Three charge contexts, each with its own protections:

Context 1 — PM-add

Stripe SetupIntent · cardholder on-session

Radar (strong) + PM-add breaker (5/day per account).

Context 2a — Flex charge

off_session MIT · anonymous sender ≠ payer

Account Budget + per-shipment cap + distinct Radar-block routing. Radar still scores the payer on every charge.

Context 2b — Full-label

on-session CIT · payer = printer

Radar (strong) + Account Budget.

The asymmetry: 2a is where SendMo carries the risk itself, because Stripe Radar can't see the anonymous sender on an off-session merchant-initiated transaction. Contexts 1 and 2b are ordinary Stripe surfaces — Radar handles them.

3. How to work with it

For admins (raising a budget)

  1. Sign in to SendMo as the admin account.
  2. Go to /admin and expand the Set Account Budget panel.
  3. Paste the target user's UUID, the new daily $ amount, the new weekly $ amount, click Set. Success or server error appears inline.
  4. Alternative (no UI): from Supabase Studio or any authenticated supabase-js client — supabase.rpc('set_account_budget', { target_user_id, daily_cents, weekly_cents }).

The RPC is admin-gated server-side (SECURITY DEFINER + role check). A column-level REVOKE UPDATE on the budget columns prevents any user from self-raising via PostgREST, even though the "Users can update own profile" RLS policy would otherwise allow it. Both gates are required because Supabase RLS doesn't override column grants — together they form a defense-in-depth lock.

For agents and devs working on payments

Do NOT (per PAYMENTS.md §9):

  • Bypass checkAccountBudget() in labels/ or payments/. It runs before PI creation so a refusal never leaves a charged-but-no-label race.
  • Treat a Stripe Radar block as a card decline in stripe-webhook/. Use the radar_blocked branch — payer's card is fine, link must stay Active.
  • Bypass the per-shipment cap. display_price_cents is server-derived from EasyPost, never trusted from the client.
  • Write to transactions outside stripe-webhook/ (Rule 16; the lone exception is the comp_grant path in labels/).

Observability — these event types appear in event_logs:

  • velocity.limit_hit — budget or PM-add breach. properties.layer = account_budget | pm_add.
  • label.flex_radar_blocked — synchronous Radar-block hint (from labels/).
  • stripe.radar_blocked — authoritative Radar block (from stripe-webhook/, severity warn).
  • stripe.radar_check_failed — charge fetch failed during the Radar check; conservative fall-through to the decline path.
  • stripe.radar_block_email_failed, velocity.budget_email_failed — outbound notification failures.

The radar_blocked row in link_state_events is the audit record for each block.

4. Tests — what runs in CI on every push

The "Provide Tests" GitHub Actions workflow runs automatically on every push and PR to main. No special invocation needed when shipping risk-intel changes. Per TESTING.md:

  • npm run test:unit — Vitest, all tests/unit/*. Includes tests/unit/budget.test.ts (16 tests: window math, fail-open, per-mode, defaults, daily-vs-weekly precedence). Blocking.
  • npm run test:e2e — Playwright mocked suite. Includes tests/e2e/account-budget-admin.spec.ts (3 tests for the Admin UI). Currently non-blocking (continue-on-error) while the e2e suite stabilises overall — make it blocking once health is reliably green.
  • tsc -b — blocking.

Pushing changes to risk-intel code: the unit + e2e specs above run automatically. Confirm green before declaring done. Integration tests (npm run test:integration) and real-service e2e are excluded from the default (they cost real EasyPost / Stripe API calls) — run them deliberately when meaningful.
